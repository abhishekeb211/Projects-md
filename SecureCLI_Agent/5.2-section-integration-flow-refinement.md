## Section Integration Approach

### Content Conversion Strategy

Phase 4 markdown sections are converted to LaTeX format following these principles:

1. **Markdown → LaTeX Mapping:**
   - Headers: `##` → `\subsection{}`, `###` → `\subsubsection{}`
   - Bold: `**text**` → `\textbf{text}`
   - Code: `` `code` `` → `\code{code}`
   - Citations: `[1]` → `\cite{ref:label}`
   - Math: `κ = 0.78` → `$\kappa = 0.78$`
   - Code blocks: JSON/Rego → `\begin{verbatim}...\end{verbatim}` or `lstlisting` environment

2. **Acronym Consistency:**
   - All acronyms use custom commands from `macros.tex`
   - First occurrence defines acronym: `\sast{} (Static Application Security Testing)`
   - Subsequent uses: `\sast{}` (without expansion)

3. **Mathematical Notation:**
   - Finding tuple: `\finding` (defined as `\mathbf{f}`)
   - Policy function: `\policy` (defined as `\pi`)
   - Triage function: `\triage` (defined as `\tau`)
   - MTTR: `\mttr{}` (abbreviation with `\text{}`)

4. **Placeholder Preservation:**
   - Experimental results: `\result{E1}{TBD}` for pending data
   - TODO items: `\todo{description}` for actions required

### Section File Structure

Each section file (`sections/XX-section-name.tex`) follows this structure:

```latex
% Section X: Section Title
% Converted from 4.X-section-name.md
% Module 5.2: Section Integration & Flow Refinement

\subsection{Subsection Title}
Content here...

% Transition paragraph to next section (if applicable)
Transition text referencing current section and previewing next section.
```

## Flow Checks

### RQ Tracking

Research questions (RQ1-RQ5) are introduced in Section 1 (Introduction) and tracked throughout the paper:

- **Section 1 (Introduction):** RQs introduced in subsection 1.5 with one-sentence motivation each
- **Section 3 (Methodology):** RQs mapped to experiments (E1-E4) and evaluation framework
- **Section 6 (Evaluation):** Each RQ addressed with quantitative/qualitative results
- **Section 7 (Discussion):** RQ answers summarized in subsection 7.1 with evidence synthesis
- **Section 8 (Limitations):** RQ limitations acknowledged (construct validity, statistical power)

**Quality Check:** All RQs appear in ≥3 sections (Introduction, Methodology/Evaluation, Discussion).

### Figure/Table References

Figure and table references follow IEEE style:

- **Reference format:** `Figure~\ref{fig:label}` and `Table~\ref{tab:label}`
- **Reference order:** Figures/tables referenced before appearance (forward references only)
- **Caption format:** Captions precede tables, follow figures

**Placeholder figures/tables:**
- Figure X: Tool-switching time histogram (Section 3.1)
- Table 3: MTTR Reduction (Section 6.1) - `\result{E1}{TBD}`
- Table 4: Triage Accuracy (Section 6.2) - `\result{E2}{TBD}`
- Table 5: Security Debt Velocity (Section 6.3) - `\result{E3}{TBD}`

**Quality Check:** All figures/tables have captions and are referenced in text before appearance.

### Citation Consistency

Citations follow IEEE numeric format:

- **First mention:** Full author name with year, e.g., "Smith (2020) documents..."
- **Subsequent mentions:** Author name or number, e.g., "Smith~\cite{ref:smith2020}" or "[1]"
- **Bibliography:** All `\cite{}` references must exist in `refs.bib`

**Citation mapping:**
- `[1]` → Forsgren et al. (2022) DORA
- `[2]` → Smith (2020) SARIF
- `[3]` → Brown & Liu (2023) Privacy-preserving AI
- `[4]` → Forsgren et al. (2022) DORA Metrics
- `[5]` → Zhang (2024) Local vs. Cloud LLMs
- `[6]` → OWASP Top 10 for LLMs (2023)
- `[7]` → Hevner et al. (2004) DSR

**Quality Check:** All `[n]` placeholders in markdown converted to `\cite{ref:label}` with corresponding BibTeX entries.

## Consistency Audit

### Notation Consistency

Mathematical notation uses `\newcommand` definitions from `macros.tex`:

- **Finding tuple:** `\finding` (not `f`, `finding`, or `\mathbf{f}` in text)
- **Policy function:** `\policy` (not `π`, `pi`, or `policy`)
- **Triage function:** `\triage` (not `τ`, `tau`, or `triage`)
- **MTTR:** `\mttr{}` (not "MTTR", `MTTR`, or "Mean Time To Remediate" after first definition)

**Audit Result:** All mathematical symbols consistently use macro definitions; no manual formatting.

### Acronym Consistency

Acronyms use custom commands and follow first-use expansion pattern:

- **First occurrence:** `\sast{} (Static Application Security Testing)` or `\sarif{} (Static Analysis Results Interchange Format)`
- **Subsequent uses:** `\sast{}`, `\sarif{}`, `\pac{}`, `\llm{}`, `\opa{}`, `\dast{}`

**Acronym definitions:**
- SAST: Static Application Security Testing
- DAST: Dynamic Application Security Testing
- SARIF: Static Analysis Results Interchange Format
- PaC: Policy-as-Code
- OPA: Open Policy Agent
- LLM: Large Language Model
- MTTR: Mean Time To Remediate

**Audit Result:** All acronyms consistently use macro commands; first occurrence expands full term.

### Capitalization Consistency

Capitalization follows these rules:

- **Platform name:** "AegisCLI" (capital A, capital C, capital L, capital I)
- **Generic reference:** "the platform", "our system", "AegisCLI" (when referring to the platform itself)
- **Component names:** "Scanner Adapter", "Triage Engine", "Policy Engine" (proper nouns)
- **Generic terms:** "scanner", "triage", "policy" (lowercase when generic)

**Audit Result:** "AegisCLI" consistently capitalized; generic terms lowercase unless referring to specific components.
